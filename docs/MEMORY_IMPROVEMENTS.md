# Улучшение долговременной векторной памяти

Идеи для более качественной и «крутой» работы с долговременной памятью: приоритеты, что менять в коде, и как измерить эффект.

---

## Текущее состояние (кратко)

- **Сохранение:** явные «Запомни…», quickFactCheck (LLM), отложенный анализ диалогов (пары user→bot).
- **Хранение:** Qdrant, коллекции по доменам (work, family, personal…), один вектор на факт (content + metadata).
- **Поиск:** один домен по классификации → векторный поиск по сообщению пользователя; при малом числе результатов — кросс-доменный поиск, limit 5.
- **Контекст для агента:** «Релевантный контекст из долговременной памяти (домен: X): …» — только текст фактов, без весов/даты.

---

## 1. Дедупликация и обновление фактов (высокий приоритет)

**Проблема:** один и тот же факт может сохраняться много раз («жена Юля», «мою жену зовут Юля» и т.д.), засоряет память и поиск.

**Идея:**
- Перед сохранением делать поиск по тому же домену с тем же запросом (или по embedding нового факта).
- Если есть очень похожий факт (score > 0.92–0.95): не создавать новую точку, а обновить payload существующей (например, обновить `content` на более полную формулировку, обновить `timestamp`, слегка поднять `importance`).
- Опционально: хранить в payload `source: "explicit" | "extracted"` и при обновлении не затирать явные факты менее точными.

**Где менять:**  
`enhancedDomainMemory.saveMemory()` — перед вызовом `vectorService.saveMemory()` вызывать `searchMemories` по `content` в том же `domain`, проверять max score; при высоком сходстве — `update` (или upsert с тем же id) вместо создания новой точки.  
В Qdrant: либо поиск по своей коллекции + update по id, либо использовать id от «похожего» факта при upsert.

**Эффект:** меньше дубликатов, чище контекст, меньше рост размера БД.

---

## 2. Мультизапрос при поиске (средний приоритет)

**Проблема:** один запрос = один embedding сообщения пользователя. Короткие или неоднозначные сообщения дают слабый семантический поиск.

**Идея:**
- Строить 2–3 варианта запроса: исходное сообщение; короткий «суть» (через LLM или эвристики: убрать стоп-слова, «как зовут жену» → «жена имя»); для личных тем — обобщённый запрос («семья», «близкие»).
- Искать по каждому варианту, объединять результаты по id, сортировать по max(score) или по взвешенной сумме.
- Либо один раз эмбеддить конкатенацию вариантов (если API позволяет batch) и искать по одному вектору (усреднение или конкатенация векторов).

**Где менять:**  
`QdrantVectorService.searchMemories` / слой в `getDomainContext`: перед поиском генерировать дополнительные query-варианты; выполнять несколько поисков; мержить и дедуплицировать по id.

**Эффект:** лучше находятся факты при коротких или разговорных формулировках («а как зовут Юлю?», «что насчёт жены»).

---

## 3. Учёт важности и давности при ранжировании (средний приоритет)

**Проблема:** сейчас порядок определяется только score похожести. Важные и недавние факты могут теряться.

**Идея:**
- В payload уже есть `importance` и `timestamp`. При равном или близком score (например, разница < 0.05) пересортировать по комбинированной метрике, например:  
  `finalScore = similarityScore * (0.7 + 0.2 * importance) * recencyFactor`,  
  где `recencyFactor` — убывающая функция от возраста (например, за последние 7 дней = 1, дальше плавное снижение).
- Реализовать либо в коде после `search()` (пересортировка), либо хранить в Qdrant precomputed weight и использовать при сортировке (если поддерживается).

**Где менять:**  
После `this.client.search()` в `QdrantVectorService.searchMemories` и в `searchAllDomains`: для каждого результата вычислить `recencyFactor` по `timestamp`, объединить с `score` и `importance`, отсортировать заново.

**Эффект:** в контекст чаще попадают действительно важные и актуальные факты.

---

## 4. Несколько «якорных» фактов в каждом ответе (средний приоритет)

**Проблема:** контекст полностью завязан на семантику текущего сообщения. Глобально важные вещи (имя, жена, работа) могут не попасть, если сообщение про другое.

**Идея:**
- Хранить в payload флаг или отдельное поле `is_anchor` / `priority: "high"` для явно сохранённых и очень важных фактов.
- При сборке контекста для агента: всегда добавлять 1–3 «якорных» факта (по importance или по тегу), плюс 3–5 по векторному поиску по сообщению. Итого ограничить общий лимит (например, 8 фактов или 1500 символов), чтобы не раздувать промпт.

**Где менять:**  
`saveMemory`: для явных «Запомни» ставить в payload `isAnchor: true` или `priority: "high"`.  
`getDomainContext` / слой над поиском: один запрос «топ по importance в личном/семья», второй — обычный векторный поиск по сообщению; объединить, убрать дубли, обрезать по лимиту.

**Эффект:** Кира стабильно «помнит» ключевые вещи (имя жены, работа и т.д.) даже в отвлечённых диалогах.

## 5. Нормализация формулировки факта перед сохранением (низкий/средний приоритет)

**Проблема:** сохраняем сырой текст («мою жену зовут Юля»). Поиск по «как зовут жену» может быть чуть хуже из-за разницы формулировок.

**Идея:**
- Перед сохранением прогонять факт через короткий LLM-шаг: «Приведи к одной короткой нейтральной формулировке факта о пользователе, без местоимений „мой/моя“, сохрани имена и цифры.»  
  Пример: «мою жену зовут Юля» → «Жену пользователя зовут Юля».
- Сохранять в векторную БД уже нормализованную строку; опционально в payload хранить и исходную (для отображения пользователю).

**Где менять:**  
`enhancedFactExtraction`: для явных фактов и для результатов quickFactCheck вызывать нормализацию (отдельная функция с вызовом LLM); в `saveMemory` передавать уже нормализованный `content`.

**Эффект:** более стабильный поиск и меньше дубликатов при разных формулировках.

---

## 6. Расширение контекста памяти в промпте агента (низкий приоритет)

**Проблема:** сейчас контекст — плоский блок текста. Агент не видит, что важнее, что свежее.

**Идея:**
- В `buildMemoryContextBlock` передавать не только текст, но и структуру, например:  
  «Важные факты: …; По текущей теме: …» или «[высокая важность] …; [по запросу] …».
- Либо добавлять короткие префиксы к каждому факту: «(важно) Жену зовут Юля», «(недавно) …».

**Где менять:**  
`agentMemoryContext.buildMemoryContextBlock`, формат строки, которую подставляют в промпт conversationAgent.

**Эффект:** модель лучше расставляет приоритеты при использовании памяти.

---

## 7. Команды для пользователя: что помнишь / удалить (низкий приоритет)

**Идея:**
- «Что ты помнишь обо мне?» / «Какие факты хранишь?» — поиск по домену «personal» или по нескольким доменам, вывод последних N фактов (без эмбеддинга, по timestamp или по getRecentMemories).
- «Забудь что …» / «Удали из памяти …» — поиск по запросу, удаление найденных точек по id (в Qdrant есть delete).

**Где менять:**  
Новые команды в `memoryCommands.ts` или `commands.ts`; в QdrantVectorService при необходимости добавить метод удаления по фильтру (content similarity или по id после поиска).

**Эффект:** прозрачность и контроль для пользователя, меньше ощущение «чёрного ящика».

---

## 8. Периодическое сжатие старых фактов (долгосрочно)

**Идея:** раз в N дней для каждого домена: выбирать старые факты (например, старше 30 дней), отправлять их в LLM для суммаризации в 1–2 предложения, сохранять суммаризированный факт как новую точку, старые точки удалять или помечать «архив». Так долговременная память не растёт бесконечно и остаётся читаемой.

**Где менять:** отдельный скрипт или фоновый воркер; вызовы поиска по домену + фильтр по timestamp, затем LLM + save + delete.

---

## Рекомендуемый порядок внедрения

1. **Дедупликация при сохранении** — быстро снижает шум и рост БД.  
2. **Якорные факты + учёт importance/recency при поиске** — заметно улучшают релевантность контекста.  
3. **Мультизапрос или нормализация формулировок** — следующий шаг для качества поиска.  
4. Остальное — по желанию и времени.

Если нужно, можно начать с пункта 1 (дедупликация) и расписать конкретные патчи по файлам и функциям.
